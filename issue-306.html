Teams & Session Scoping

Teams as membership containers: team model, group auto-grant, and session scoping

#306 enhancement depends on: #302, #304, #305

Summary

Teams are a first-class concept above UserGroup (#302). A Team is a named group of people that grants UserGroup memberships to its members. This provides two capabilities the base RBAC model cannot express:

  1. Bulk group assignment — add a user to a team, they get all groups linked to that team. Remove them, the groups go away (unless directly assigned).
  2. Session-scoped access — team memberships can be time-limited and session-scoped, so a cabin lead for Session A automatically loses access when Session A ends.

Sixth building block of the affiliate RBAC redesign (#300). Depends on #302 (models) and #304 (permission checks). The Teams UI tab in the #305 mockup is implemented by this issue.

Why Teams?

The UserGroup → PermissionSet → Permission model from #301-#305 only supports direct user-to-group assignments. Real camp operations have patterns that don't fit:

Scenario Today (without teams) With teams
Session cabin leads Give housing.write permanently, or don't give it at all Session-scoped team grants Housing Coordinator group; membership auto-expires
Staffing committee Each person individually added to the group Add everyone to the team once; team auto-grants the group
Multi-hat volunteers Multiple direct group assignments, manually tracked Direct group + team membership; effective permissions are the union
Medical team rotation Permanently assigned, manually removed later Session-scoped team; access auto-expires with the session

Data Model

Team

class Team(TimeStampedModel):
    name = models.CharField(max_length=100)
    affiliate = models.ForeignKey(Affiliate, on_delete=models.CASCADE, related_name="teams")
    description = models.TextField(blank=True)
    is_system = models.BooleanField(default=False)
    session = models.ForeignKey("apps.sessions.Session", null=True, blank=True, on_delete=models.CASCADE)
    expires_at = models.DateTimeField(null=True, blank=True)  # null = no expiration

    class Meta:
        constraints = [
            UniqueConstraint(fields=["name", "affiliate"], name="unique_team_name_per_affiliate"),
        ]
  • session is optional. If set, the team is session-scoped. If null, it's a standing team.
  • expires_at is optional. Can be set independently of session.
  • is_system prevents deletion (no system teams shipped by default, reserved for future).

TeamGroupLink

class TeamGroupLink(TimeStampedModel):
    team = models.ForeignKey(Team, on_delete=models.CASCADE, related_name="group_links")
    user_group = models.ForeignKey(UserGroup, on_delete=models.CASCADE, related_name="team_links")

    class Meta:
        constraints = [
            UniqueConstraint(fields=["team", "user_group"], name="unique_team_group_link"),
        ]

A team links to one or more UserGroups. When a user joins the team, they receive membership in all linked groups. When they leave, team-granted memberships are removed (unless the user also has a direct membership).

TeamMembership

class TeamMembership(TimeStampedModel):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    team = models.ForeignKey(Team, on_delete=models.CASCADE, related_name="memberships")
    is_lead = models.BooleanField(default=False)
    assigned_by = models.ForeignKey(settings.AUTH_USER_MODEL, null=True, blank=True, on_delete=models.SET_NULL)

    class Meta:
        constraints = [
            UniqueConstraint(fields=["user", "team"], name="unique_team_membership"),
        ]

is_lead grants the team lead the ability to manage their own team's membership (add/remove members) without needing staff.manage.

Permission Resolution

The effective group set for a user becomes:

effective_groups(user, affiliate) = direct_groups(user, affiliate) ∪ team_granted_groups(user, affiliate)

Where team_granted_groups resolves: for each active TeamMembership (not expired), find the team's TeamGroupLinks, and collect the linked UserGroups.

The permission check (user_has_affiliate_permission from #304) is updated to include team-granted groups in its resolution path. This is a change to the access layer, not a new decorator.

Expiration Handling

  • expires_at timestamp — checked at query time. Expired memberships are filtered out. No celery task needed for correctness; a periodic cleanup task can remove expired rows for hygiene.
  • session FK — when a Session is closed or archived, its teams' memberships are effectively inactive: team.session_id is None OR team.session.status != 'CLOSED'.

These routes are additive to #305's route table. #305 covers groups/permission-sets/permissions management UI; this issue covers the teams UI.

Team Lead Permissions

Team leads (users with TeamMembership.is_lead=True) can:

  • Add and remove members from their own team
  • View their team's group links

Team leads cannot:

  • Edit the team's name, description, or group links (requires staff.manage)
  • Delete the team (requires staff.manage)
  • Create new teams (requires staff.manage)
  • Manage other teams (requires staff.manage)

This is a relationship-based check (TeamMembership.is_lead=True AND team == the_team_in_question), not a PermissionSet permission. This avoids needing a new catalog permission for what is fundamentally a scoped ownership check.

What This Issue Does NOT Cover

  • Cabin-level scoping — scoping access to specific cabins within a session. Requires a cabin FK on TeamMembership or a separate scoping model. Explicitly deferred.
  • Team permission sets — teams grant groups, not permission sets directly.
  • Cross-affiliate teams — teams are scoped to a single affiliate.
  • Team inheritance — no nesting of teams within teams.

Open Questions

We want your input
  1. Session scoping vs membership scoping — This issue proposes team-level scoping: the team is session-scoped, and all memberships within it inherit that scope. Is this the right level?
  2. Should team leads be a PermissionSet permission? Currently proposed as a relationship check (is_lead). The alternative is adding team.lead to the permission catalog. Which is cleaner?
  3. Expiration cleanup — Should expired TeamMembership rows be cleaned up by a periodic task, or just filtered at query time?
Security Considerations
  • Team-granted groups are resolved at query time — no stale access after expiration
  • Team leads can only manage their own team's membership, not edit team structure
  • All team operations are audit-logged via log_action()
  • Cross-org isolation enforced — no access to other affiliates' teams
  • Central superadmins retain escalation into any affiliate's teams
  • Direct group memberships are preserved when a team-granted group is removed — no accidental privilege loss
