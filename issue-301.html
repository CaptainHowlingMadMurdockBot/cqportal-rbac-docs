Permission Catalog

Granular capability registry — the foundation of the RBAC redesign

#301 enhancement depends on: none

The Problem

Currently access control is binary — you hold a role string (AFFILIATE_ADMIN) or you don't. There is no way to express "can read housing but not write" or "can approve volunteers but not manage BGC" without adding a new role for every combination. The result is either over-granted access (give full admin because a narrow capability is needed) or under-granted access (no path exists at all). The flat role model cannot compose.

The Solution

A Python-defined permission registry in apps/tenancy/permissions.py. Permissions are constants grouped by domain. The registry is the single source of truth: views and decorators check against it, and new permissions require code review and a PR.

python 
# apps/tenancy/permissions.py

HOUSING_READ      = "housing.read"
HOUSING_WRITE     = "housing.write"
HOUSING_AUTO_ASSIGN = "housing.auto_assign"

PERMISSION_CATALOG = {
    "housing": {
        HOUSING_READ: "View cabin assignments and housing board",
        HOUSING_WRITE: "Create, edit, and delete cabin assignments",
        HOUSING_AUTO_ASSIGN: "Run automatic cabin assignment algorithm",
    },
    # ... more domains
}

Every permission is a domain.action string. Domains are the top-level grouping (housing, volunteers, registrations, etc.) and actions are the granular capability within that domain (.read, .write, .auto_assign). The catalog maps each permission constant to a human-readable description so the UI and audit logs can show what a permission actually does.

Design Decisions

1. Python-defined, not database-driven

Permissions ship with code releases, not as DB rows. This is safer — a misconfigured database entry can never grant a capability that wasn't already reviewed in a PR. The catalog lives in version control, diffed and discussed like any other code.

2. Domain-prefixed naming

Every permission follows the domain.action format. This makes permissions grep-able: a search for housing. finds every housing capability across the codebase, and a search for .read finds every read permission. The prefix is the domain, the suffix is the action.

3. Read/write separation

Every domain that has writable state exposes both .read and .write. This is the minimum viable separation: a role can always be granted read-only access to a domain without write. Additional manage-tier actions (like .auto_assign or .delete) layer on top of the read/write baseline.

4. Board governance excluded

Board access is a governance relationship modeled by BoardMember, not an operational permission. Board members don't get staff permissions; they get board-specific capabilities through a separate model. Keeping the catalog focused on operational access prevents the two concerns from leaking into each other.

Questions for the Community

We want your input on the catalog
  • Are there any permissions missing from the catalog?
  • Is the domain grouping intuitive, or should domains be reorganized?
  • Should medical.write be split into medical.write_records and medical.write_emergency?
  • Are there affiliate-specific permissions that should be in the core catalog?

These questions are open. The catalog is the foundation every later issue builds on, so getting the permission vocabulary right now saves rework later. If a capability is missing or a domain boundary feels wrong, now is the time to say so.

Security Considerations
  • Permissions ship with code releases, not DB-configurable. A database edit cannot create or widen a permission — only a code change, reviewed and merged, can.
  • A misconfigured DB permission could grant unintended access — this is why the catalog is Python-defined. The database stores assignments (who has which permission set), not definitions (what a permission does).
  • New permissions require code review and PR approval. Adding a permission means adding a constant to permissions.py, which is visible in the diff and subject to the same scrutiny as any access-control change.
  • The catalog is the single source of truth. Views check against it; a permission string that isn't in the catalog is rejected, not silently accepted.
