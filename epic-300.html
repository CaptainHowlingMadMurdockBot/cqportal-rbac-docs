Affiliate RBAC Redesign

UserGroups, PermissionSets, and granular permissions

Epic #300 enhancement

Why This Redesign

The current system has only 2 affiliate roles: AFFILIATE_ADMIN and AFFILIATE_VOLUNTEER_MANAGER. Originally there were 4 roles, but migration 0036 collapsed AFFILIATE_REGISTRAR and AFFILIATE_MEDICAL_STAFF because the flat model couldn't express real-world org charts. Two roles cannot capture the nuance of camp operations where staff wear different hats and need scoped, least-privilege access.

Real-world consequences of the flat role model
  • A camp director who needs cabin assignments must be given AFFILIATE_ADMIN, which also grants community moderation, library deletion, and full reports — violating least privilege.
  • A registrar who needs to see housing assignments to coordinate roommates has no read-only option — it's full admin or nothing.
  • A camp nurse who needs health records and rosters gets AFFILIATE_ADMIN as the only path, exposing them to financial data, BGC status, and volunteer management they shouldn't see.
  • Volunteers who wear multiple hats (camp director + board president) can't cleanly separate operational access from board governance.

The Team & Roles UI at /o/<org>/a/<slug>/team/ will be evolved, not replaced. The redesign layers a flexible permission model on top of the existing affiliate infrastructure rather than introducing a parallel system.

Architecture

The redesign introduces a three-tier model that maps users to granular permissions through affiliate-scoped groups and named permission sets.

User
A person with an account
↓
UserGroup
Affiliate-scoped grouping of members
↓
PermissionSet
Named bundle of permissions
↓
Permission
Granular capability (e.g. housing.read)

What Stays the Same

The redesign is additive. The following roles and structures remain as RoleAssignment and are untouched by the new model:

  • PARENT — stays as RoleAssignment
  • VOLUNTEER — stays as RoleAssignment
  • PLATFORM_ADMIN — stays as RoleAssignment
  • CENTRAL_* roles — stay as RoleAssignment
  • Board governance (BoardMember) — stays separate from the permission system

Default Affiliate Template

When a new affiliate is provisioned, 8 default groups are auto-created from a template. Each group bundles a set of permissions appropriate to a common camp role.

Group Key permissions
Affiliate Admin library.read, sessions.read, reports.* (oversight)
Camp Director housing.*, sessions.*, events.*, volunteers.read, forms.*, reports.*, community.moderate, staff.manage
Volunteer Coordinator volunteers.*, reports.view_volunteer, reports.view_training, community.post
Housing Coordinator housing.*, sessions.read
Registrar registrations.*, forms.*, sessions.read, housing.read
Medical Staff medical.*, sessions.read, reports.view_attendance
Community Moderator community.moderate, community.manage_sections
Staff Viewer *.read across all domains

Design Principles

1. Restricted by default

Affiliates expand permissions, not contract. The core template starts narrow; affiliates add groups or permission sets to grant more access, never to remove it.

2. Python-defined permissions

Permissions ship with code, not DB-configurable. This keeps the catalog under version control and subject to code review, preventing accidental privilege escalation through a database edit.

3. Unified UI

Evolve the existing Team & Roles page, no parallel system. The affiliate staff management experience stays in one place; the new group/permission model is surfaced through the page affiliates already use.

4. Board governance stays separate

BoardMember is unchanged. Board access is a governance relationship, not an operational permission, so it remains orthogonal to the permission model.

5. Migration guarantee

No access lost, no access gained. Every existing AFFILIATE_ADMIN keeps full access; every AFFILIATE_VOLUNTEER_MANAGER keeps their volunteer management access. The migration is a re-expression of current state, not a re-granting.

6. Org isolation enforced at model level

The chain GroupMembership → UserGroup → Affiliate → Organization means a permission check can never silently cross an org boundary. Isolation is structural, not a convention.

Security Considerations
  • Permissions are Python-defined, not database-driven — prevents misconfiguration granting unintended access. A database edit can never create a new permission or widen an existing one.
  • Org isolation enforced at model level — GroupMembership → UserGroup → Affiliate → Organization. The permission check traversal can never cross an org boundary.
  • Self-lockout prevention — you cannot remove your own last group with staff.manage. The system blocks the action before it commits.
  • Central role escalation preserved — CENTRAL_SUPERADMIN passes all checks in the same org, so central staff retain their override capability.
  • No access lost or gained during migration — the migration is a faithful re-expression of the current role assignments into the new group/permission model.
