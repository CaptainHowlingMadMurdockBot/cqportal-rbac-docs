Access Refactor

Replace role-string checks with permission-based checks across all views

#304 enhancement depends on: #301, #302, #303

The Problem

Currently access.py functions like user_has_affiliate_role(user, affiliate) check for role strings (AFFILIATE_ADMIN, AFFILIATE_VOLUNTEER_MANAGER). Decorators like @require_affiliate_role pass if the user holds any affiliate role. There is no way to gate a view on a specific capability (e.g., "can write housing") without hardcoding role checks directly in the view body — which scatters access logic across the codebase and makes it impossible to reason about who can do what from a single place.

New Access Functions

The refactor introduces permission-aware functions that check the UserGroup → PermissionSet → Permission chain instead of flat role strings. These become the new building blocks for all affiliate-scoped access decisions.

python 
def user_has_affiliate_permission(user, affiliate, permission: str) -> bool:
    '''True if the user holds the given permission on this affiliate
       (via any group membership), OR has a central role in the same org.'''
    ...

def user_has_any_affiliate_permission(user, affiliate, *permissions) -> bool:
    '''True if the user holds at least one of the given permissions.'''
    ...

And the new decorator:

python 
@require_affiliate_permission("housing.write")
def affiliate_cabin_assign(request, slug):
    ...

Decorator Migration Table

Every existing @require_affiliate_role decorator and the _require_affiliate_admin() guard need to be mapped to a specific permission. This is the full set of old-to-new mappings:

Old New
@require_affiliate_role (on volunteer views) @require_affiliate_permission("volunteers.read")
@require_affiliate_role (on housing views) @require_affiliate_permission("housing.write")
@require_affiliate_role (on report views) @require_affiliate_permission("reports.view_*") (per-report)
@require_affiliate_role (on session views) @require_affiliate_permission("sessions.write")
@require_affiliate_role (on event views) @require_affiliate_permission("events.write")
_require_affiliate_admin(request) (team management) @require_affiliate_permission("staff.manage")

Before/After Code Example

The same view, gated the old way and the new way. The old decorator let both affiliate roles through with no granularity. The new decorator restricts access to exactly the capability the view requires.

Before
python 
@require_affiliate_role
def affiliate_cabin_assign(request, slug):
    # Both AFFILIATE_ADMIN and AFFILIATE_VOLUNTEER_MANAGER pass
    # No way to restrict to housing-specific access
    ...
After
python 
@require_affiliate_permission("housing.write")
def affiliate_cabin_assign(request, slug):
    # Only users with housing.write permission pass
    # Camp Director, Housing Coordinator pass
    # Volunteer Coordinator, Registrar (read-only) do not
    ...

Central Role Escalation

Central roles escalate into all affiliates

Central roles (CENTRAL_SUPERADMIN) continue to escalate into all affiliates within the same org. Permission check functions detect central RoleAssignment rows and grant all permissions — same as current behavior. No view-level special-casing is needed; the escalation happens inside user_has_affiliate_permission itself.

What Stays Unchanged

Not every access check changes. The following decorators and access models remain exactly as they are today — they are orthogonal to the affiliate permission model and operate on different relationships:

  • @require_central_role — central roles stay as RoleAssignment
  • @require_parent / @require_volunteer — end-user roles stay as RoleAssignment
  • @require_platform_admin — unchanged
  • Board decorators (@require_board_member, @require_board_chair_or_secretary) — unchanged, board has its own access model via BoardMember

View Audit

Every view currently decorated with @require_affiliate_role needs to be audited and mapped to the correct permission. There are approximately 30+ affiliate-scoped views spread across the following files:

  • apps/web/views.py
  • apps/compliance/views.py
  • apps/housing/views.py
  • apps/camp/views.py
  • apps/events/views.py
  • apps/volunteers/views.py
  • apps/training/views.py

Each view must be examined, its domain and action determined, and the correct permission string applied. The migration table above is the starting point, but the audit is what confirms the mapping is right for every individual view.

Questions for the Community

We want your input on the migration
  • Are the permission mappings in the table correct? Should any views have different permissions?
  • Should report views be gated on individual report permissions (reports.view_volunteer, reports.view_registration) or a broader reports.read?
  • Should the volunteer list view require volunteers.read or volunteers.write? (Currently both admin roles can see it.)
  • Are there affiliate-scoped views we missed in the audit?
Security Considerations
  • access.py remains the single place for permission checks
  • No raw RoleAssignment.objects.filter(role__in=AFFILIATE_ROLES) queries outside access.py
  • Central role escalation preserved — same org access for central roles
  • Cross-org isolation maintained — user in org A gets no permissions in org B
  • Permission checks are per-view, not per-role — least privilege enforced at the code level
