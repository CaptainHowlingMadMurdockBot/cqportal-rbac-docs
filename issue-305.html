Team & Roles UI Evolution

Evolve the existing UI to manage groups, permission sets, and memberships

#305 enhancement depends on: #301, #302, #303, #304

The Problem

The current Team & Roles page lets an affiliate admin pick from two roles (AFFILIATE_ADMIN, AFFILIATE_VOLUNTEER_MANAGER) and assign them to users by email. The template still has badge colors and descriptions for the collapsed AFFILIATE_MEDICAL_STAFF and AFFILIATE_REGISTRAR roles — a visible artifact of the flat model straining under real-world needs. After the RBAC redesign (#301-#304), the underlying model is UserGroup → PermissionSet → Permission. The UI needs to evolve.

Before/After UI Mockup

The current page is a simple role picker. The evolved page exposes groups, effective permissions, and a permission catalog — all without leaving the affiliate admin section.

Evolved Page Structure

The single Team & Roles page expands into four sub-pages, each focused on a specific management task:

  • /o/<org>/a/<slug>/team/ — Staff table with groups as badges, effective permissions summary
  • /o/<org>/a/<slug>/team/groups/ — Group list, create custom groups, rename, assign permission sets
  • /o/<org>/a/<slug>/team/permission-sets/ — Permission set CRUD, create custom sets from catalog
  • /o/<org>/a/<slug>/team/permissions/ — Read-only permission catalog display

Access Control

  • Only users with staff.manage permission (Camp Director by default per #303) can access group/permission set management
  • All affiliate staff (any group membership) can view the staff table
  • Central superadmins can manage any affiliate's groups
  • Self-lockout prevention: cannot remove your own last group with staff.manage

UI Design

  • Bootstrap 5 cards/tables consistent with existing affiliate admin pages
  • HTMX for add/remove member and group operations (no full page reload)
  • Permission picker: checkbox grid grouped by domain, read/write pairs side by side
  • Confirmation modals for destructive actions
  • Audit log entries for all management actions via log_action()
  • Group badges use same visual language as current role badges (colored pills)

Questions for the Community

We want your input on the UI
  • Should the staff table show effective permissions by default, or only on expand?
  • Should the permission catalog page be visible to all staff, or only those with staff.manage?
  • Is the sub-page structure (groups/, permission-sets/, permissions/) intuitive, or should it be tabs on one page?
  • Should group badges use fixed colors per group, or let affiliates pick colors?
  • Is HTMX the right approach, or should we consider a full SPA for the team management page?
Security Considerations
  • Only users with staff.manage can access group/permission set management
  • Self-lockout prevention: cannot remove your own last staff.manage group
  • All management actions are audit-logged via log_action()
  • Cross-org isolation enforced — no access to other affiliates management views
  • Central superadmins can manage any affiliate's groups (same escalation as current)
  • No Django admin access needed for affiliate staff management
